01 · Intro

Why we built FinFlow

Apply cloud-native principles end-to-end — not lift-and-shift, not a single VM with a database.

Serverless compute · managed services · event-driven · caching · IaC · secrets-as-code · observability

Every architectural choice has a reason behind it.

02 · Problem

The problem

Most personal finance apps either charge a subscription, sell user data, or can't be self-hosted.

FinFlow's solution

An open-source, self-hostable finance tracker that runs at near-zero cost on Azure — with all data owned by the user.

Target audience

03 · Architecture · Services

Compute

⚡
Azure Functions
Consumption plan · Java 21 · Spring Boot

HTTP triggers proxy every request to an embedded Spring Boot instance on port 8081. Scales to zero when idle.

Why? Variable load — most of the day there's no traffic. Pay only for actual invocations.

🗄
Azure Cosmos DB
Serverless · 3 containers · partition /userId

Users, categories, transactions. Schemaless documents with embedded category data — no cross-container joins.

Why? Per-user data with no cross-user queries fits NoSQL perfectly.

03 · Architecture · Services

Cache & messaging

⚡
Azure Cache for Redis
@Cacheable · @CacheEvict on writes

Caches balance, monthly summary, and category breakdown. Cache key includes user ID + month + year.

Why? Aggregations over thousands of transactions are slow without caching.

📡
Azure Service Bus
Topic + 2 subscribers: audit-log · budget-alert

Every transaction write publishes an event. Subscribers consume independently with dead-letter queues for failed messages.

Why? Decouples writes from monitoring — a slow budget check can't block a transaction.

03 · Architecture · Services

Storage & security

📦
Azure Blob Storage
Receipts · SAS URLs · static website hosting

Receipt files accessed via 1-hour SAS URLs — frontend links directly to storage. Also hosts the React SPA on the $web container.

Why? Cheap, durable, and integrates natively with Functions.

🔐
Azure Key Vault
5 secrets · user-assigned Managed Identity · RBAC

App settings use @Microsoft.KeyVault(SecretUri=...) references. The Function App authenticates via Managed Identity — no credentials anywhere.

Why? Zero plain-text secrets in our deployment templates or environment.

03 · Architecture · Services

Observability & infrastructure as code

📊
Application Insights
Log Analytics workspace + Insights component

Auto-captures every request, exception, and outbound dependency call from the Function App.

Why? Production-grade observability without writing instrumentation code.

🏗
Bicep Infrastructure as Code
7 modular templates · reproducible deployments

Every resource on the previous slides — Cosmos, Functions, Redis, Service Bus, Blob, Key Vault, App Insights — declared as code with proper module dependencies.

Why? Tear down and redeploy the entire stack in twelve minutes.

03 · Architecture · Data flow

Creating a transaction — step by step

1
User submits form in React → POST /api/transactions with JWT
2
Azure Functions receives request, proxies to Spring Boot on port 8081
3
JWT validated → write to Cosmos DB
4
Redis cache invalidated for that user (@CacheEvict)
5
Event published to Service Bus topic transaction-events
6
Two subscribers consume in parallel: audit-log records the change · budget-alert recalculates monthly spend
7
Frontend reloads dashboard → red banner if budget exceeded

06 · Challenges

Technical hurdles · 1/2

Spring Boot inside Azure Functions

Problem: Java worker only supports Function<I,O> beans, not Spring MVC.

Solution: Static initializer starts Spring on port 8081; HTTP trigger proxies all requests. All existing controllers and security filters remain unchanged.

Secrets in plain text

Problem: First Bicep had Cosmos keys, JWT secret, and Redis password as plain-text app settings — visible in deployment logs.

Solution: Refactored to Key Vault module + user-assigned Managed Identity + KV references. Zero secrets in deployment templates.

CORS preflight blocked

Problem: Azure Functions intercepts OPTIONS at the platform level — empty 204 responses, browsers reject all cross-origin calls.

Solution: CORS configured at Functions platform via Bicep + az functionapp cors add.

06 · Challenges

Technical hurdles · 2/2

CI/CD region restriction

Problem: Azure Static Web Apps was not available in the francecentral region.

Solution: Pivoted to Blob Storage static website hosting + GitHub Actions az storage blob upload-batch.

Cache invalidation

Problem: First implementation cached aggregations forever — dashboards showed stale data.

Solution: @CacheEvict(allEntries = true) on every write operation guarantees freshness.

Empty Spending by Category chart

Problem: Backend filtered out transactions with no categoryId — chart was always empty for new users.

Solution: Added "Uncategorized" fallback bucket so all expenses appear regardless of categorisation.

07 · Conclusion

Advantages of cloud-native

  • Scales 0 → 100K users without code changes
  • Pay only for what you use — €26/year at low scale
  • Built-in security via Managed Identity & Key Vault
  • Built-in observability via Application Insights
  • No infrastructure to maintain — fully managed

A production-grade architecture for less than a coffee per month.

07 · Conclusion

Trade-offs we accept

  • Vendor lock-in — moving off Azure means rewriting integrations
  • Cold start latency — ~2–3s for first Java request after idle
  • Steeper learning curve than VM-based deployment
  • Distributed debugging is harder than local development
